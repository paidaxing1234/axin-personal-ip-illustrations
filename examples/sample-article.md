# 把一篇项目复盘变成个人 IP 配图资产

很多开源项目的 README 都会遇到同一个问题：项目真实做了很多事，但读者第一次打开仓库，只看到一堆文件、脚本和截图，很难立刻理解这个项目为什么值得关注。

我想解决的不是“生成一张好看的封面图”。封面图只能装饰首页，不能帮读者理解经验。真正有价值的是把文章里的判断、证据、流程和常见坑拆出来，生成几张可以复用的正文配图。

这套流程的输入很简单：一篇文章，一个可选的个人 IP 形象图，几项参数。脚本先读取文章，提取标题、段落、认知锚点和语言倾向，再生成 brief、shot list、image prompts、JSONL 任务、双语分发计划和发布 checklist。

我最在意的是证据。一个项目到底是已经跑通过，还是只有计划？有没有真实命令、截图、commit、用户可见结果？如果没有这些证据，配图就不应该把它画成已经完成。图像也要区分 done、partial 和 stub。

个人 IP 形象不能只是站在画面角落的头像。角色必须参与核心动作：整理文章、检查证据、连接中文和英文语境、把 prompt 打包、给未验证内容贴警示标签。去掉角色之后，如果画面仍然完全成立，说明这张图不合格。

中英文分发也不是翻译。中文内容更适合讲过程、复盘、踩坑和判断；英文内容更适合讲仓库结构、可复用工作流、README、Hacker News、X、Medium 和 LLM 可发现性。同一个项目经验要被两套语境消费。

最终我希望这个仓库像一条生产线：输入文章，输出可审核的配图策略和 prompt 包。用户可以先看 `analysis.md` 和 `illustration-shot-list.md`，确认方向，再把 `image-prompts.md` 里的提示词拿去逐张生成图片。

这个流程还有很多可以继续拓展的地方：把生成的图片回填到 README，把案例沉淀到 `cases/`，把 `llms.txt` 写得更清楚，或者把 `image-prompts.jsonl` 接到批量生图工具里。但第一步必须足够简单：任何开源用户都能先用示例文章三分钟跑通。
