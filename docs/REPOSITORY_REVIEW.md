# 仓库深度自审：从阿墨/海星混线到阿鑫主线

## 结论

这次仓库最大的问题不是“图丑”这一个表层问题，而是产品主线不够单一：默认角色、授权海星分支、仓库 slug、平台快照、README 示例、GEO 文档和脚本命名互相牵扯，导致用户一眼看到的是多个方向同时存在，而不是一套稳定的个人 IP 流程。

因此这次改造的原则是：该删掉删掉，只保留阿鑫主线；机器入口也改成 `axin-personal-ip-illustrations`，让文档、skill、脚本、平台规则和图片资产说同一种话。

## 审出的问题

### 1. 命名不是皮肤，是系统边界

原仓库公开文案叫阿墨，用户要求改成阿鑫后，如果只改 README 标题，就会出现：

- README 叫阿鑫，SKILL 仍叫阿墨。
- Codex 安装路径仍是旧名。
- Claude plugin 快照仍是旧 Skill。
- llms.txt 仍把旧角色当默认实体。
- 示例图路径仍引用旧文件。

这会让 agent 在真正执行时读到互相冲突的指令。个人 IP 仓库最怕这种冲突，因为角色名、动作库、风格 DNA 和 prompt 模板都会影响最终图像。

### 2. 旧分支会稀释新 IP

授权海星方向原本是一次迭代尝试，但用户连续反馈“丑”以后，它已经不适合继续占主线位置。继续保留它会有三个问题：

- 新用户不知道默认应该用阿鑫还是海星。
- 版权和授权边界会占用 README 和 GEO 的注意力。
- 旧的丑图脚本可能被下次 agent 误用，导致问题复发。

这类分支如果不是当前主线，就应该删掉或移出仓库，而不是作为“历史”挂在入口页上。

### 3. 图片资产必须跟流程一起校验

这个仓库的交付不是代码库本身，而是“agent 读完以后能生成稳定图片”。所以图片资产不是装饰，它们是测试样本。旧问题包括：

- 示例图少，覆盖不了角色锚点、正文配图、双语发布、GEO、仓库审查这些核心场景。
- README 引用的图和实际文件名耦合不足。
- 校验脚本只查文件是否存在和海星图尺寸，不能证明阿鑫主线完整。

改造后示例图至少覆盖五类用途：角色锚点、内容工坊、中英文发布、GEO 资产流水线、代码自审工位。

### 4. 多平台支持不能靠人工记忆同步

一个 skill 同时支持 Codex、Hermes、Claude Code、Cursor、Windsurf、Cline 和通用 agent 时，真正的风险是快照过期：

- `.claude-plugin/skills/...` 可能停留在旧版本。
- Cursor / Cline / Windsurf 规则可能还在指向旧路径。
- `skill-package.json` 可能引用不存在的图片。

因此 `scripts/sync-platform-packages.ps1` 和 `scripts/validate-repo.ps1` 必须承担“防遗忘”的职责，而不是只做复制。

### 5. GEO 不是堆关键词，而是消除歧义

GEO 文档的价值不是写很多搜索词，而是让 LLM 和搜索系统明确知道：

- 这个仓库是什么。
- 默认角色是谁。
- 入口文件在哪里。
- 能生成什么类型的图。
- 有哪些明确不做的东西。
- 中文和英文入口如何对应。

如果 GEO 页面继续讲旧角色或已删除分支，它反而会放大错误。

## 举一反三

### 一：角色名改变时，要从执行入口改起

优先改 `SKILL.md`、安装脚本、平台规则、llms.txt，再改 README 修饰文案。因为 agent 先执行的是 skill 和规则，不是 README 的宣传句。

### 二：失败方向不要长期挂在主线

用户明确否定的视觉方向，最多留下内部复盘，不应该继续放在主入口、示例图或生成脚本里。否则下次自动化会把失败方向当成可用资产。

### 三：图片生成仓库要把图片当测试

代码仓库检查函数和测试，图片流程仓库也要检查示例图尺寸、数量、路径、文件名和 README 引用。否则“图不存在 / 图太小 / 图还是旧角色”都会被误报成完成。

### 四：中英文切换不只是加一个 README.en.md

英文入口需要同步影响：

- README 顶部语言切换。
- `llms.txt` 的英文可读摘要。
- `docs/index.html` 的 JSON-LD 和 meta description。
- Prompt 模板里的 language mode。
- 示例图中是否出现英文或双语标签。

### 五：多平台仓库需要快照校验

只要仓库里有 `.claude-plugin`、Cursor rules、Cline rules、Windsurf rules，就要有校验脚本确认它们指向同一个主 skill。否则平台越多，错配越隐蔽。

### 六：命名兼容和彻底改名要分清

如果是公开 npm 包、生产 API 或大量用户已安装的 skill，保留旧 slug 做 alias 很有必要。但这个仓库仍处在打造个人 IP 的早期阶段，用户明确要求“该删掉删掉”，所以优先清理主线，减少兼容包袱。

## 当前验收标准

- 默认角色只剩阿鑫 / Axin。
- 主 skill 名为 `axin-personal-ip-illustrations`。
- README 和 README.en.md 都可直接理解项目。
- `llms.txt` 和 `llms-full.txt` 不再指向旧角色或已删除分支。
- 示例图至少 5 张，并真实存在。
- `scripts/generate-axin-examples-cli.ps1` 可以复跑生成示例图。
- `scripts/validate-repo.ps1` 能检查图片、双语 README、平台快照和旧引用残留。
- 安装脚本安装的是阿鑫 skill。
